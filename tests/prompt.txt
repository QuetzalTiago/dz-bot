- `cogs/database.py::_update_user_duration()`'s "atomic UPDATE, insert if 0
  rows" upsert only covers existing rows - for a **brand-new** `user_id` it's
  a check-then-act race: the hourly `update_user_durations` tick and a
  disconnect's `flush_user_duration` (or a `forget_me` delete racing a
  flush) are separate `asyncio.to_thread` calls that can both see 0 rows
  updated and both `session.add(User(id=user_id, ...))` for the same new
  primary key, raising `IntegrityError` (and silently losing whichever
  flush's credit doesn't win the commit race). Also, unlike every other
  public method in this file, `flush_user_duration` had no try/except around
  its `to_thread` call, so that IntegrityError could propagate uncaught out
  of `bot.py`'s `on_voice_state_update`. Fixed with a per-`user_id`

  all acquire around their `_update_user_duration`/`_delete_user_data` calls,
  serializing every DB write for a given user - plus a try/except around
  `flush_user_duration`'s call, matching its siblings. To reproduce the race
  in a test without mocking threads: swap the real `_update_user_duration`
  for a wrapper that records `time.monotonic()` before/after a small

  user_id via `asyncio.gather`, and assert the recorded intervals don't
  overlap (removing the lock reproduces the exact `sqlite3.IntegrityError:
  UNIQUE constraint failed: users.id` from a real concurrent run - this
  isn't just a theoretical TOCTOU, real threads via `asyncio.to_thread` do
  race on it).

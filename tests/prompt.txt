Environment / gotchas learned from past runs (read this before writing new tests):

- The system python has no pytest installed. Create a venv first:
  `python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements-dev.txt`
  Run pytest/ruff from inside that venv for the rest of the session.
- Commands are invoked as `await cog.command_name.callback(cog, ctx, ...)`, not
  `await cog.command_name(ctx, ...)` — hybrid_command wraps the function, so
  calling it directly skips discord.py's command machinery in tests.
- Use `tests/mocks.py::mock_ctx(bot, **kwargs)` for a ready-made Context mock,
  or build a lightweight `MagicMock()` with `AsyncMock()` on `.send`,
  `.message.add_reaction`, `.message.clear_reactions` when a full Context is
  overkill.
- `MagicMock(name="x")` does NOT set the mock's `.name` attribute (it sets the
  mock's repr) — assign `.name` after construction if the code under test
  reads `obj.name`.
- SQLAlchemy entities under `cogs/db/entities/` can be instantiated and
  asserted on directly, no engine/session needed — their `__init__` just
  copies dict keys onto attributes.
- To unit-test `cogs/api/youtube.py`, patch `cogs.api.youtube.yt_dlp.YoutubeDL`
  with a MagicMock configured as a context manager
  (`mock.__enter__.return_value = mock`) and set `.extract_info.return_value`.
  `DZ_DOWNLOAD_DIR` is read at import time via `os.environ.get(...)`, so if a
  test needs a custom download dir, `monkeypatch.setenv` then
  `importlib.reload(cogs.api.youtube)` before instantiating `YouTubeAPI`.
- Known past bug classes worth a regression test whenever you touch these
  areas again: (1) `cogs/utils/music/playlist.py::empty()` must treat a
  looped `current_song` with no queued songs as non-empty, or the state
  machine disconnects instead of replaying; (2) `cogs/api/youtube.py`
  `is_video_playable`/`download` must not hard-index `info["entries"][0]`
  (empty search results) or compare `info["duration"]` to an int without a
  None-check (livestreams report `duration: None`); (3) Lichess omits the
  `"opening"` key entirely for `aborted`/`noStart` games — `ChessGame.__init__`
  must use `.get("opening")`, not `["opening"]`.
- `ruff check .` is part of CI — run it before committing; it will flag
  unused imports left over from copy-pasted fixtures (e.g. importing a class
  directly when a test only uses a `importlib.reload`-obtained reference).
- Testing a `@tasks.loop`-decorated method directly: call
  `cog.the_loop.coro(cog)` to run the loop body once without starting the
  timer. If a test does call `.start()`, it must also stop the task before
  the test ends: `task = loop.get_task(); loop.cancel(); await task`
  (suppressing `asyncio.CancelledError`) — pytest-asyncio closes the event
  loop right after the test coroutine returns, so a fixture that calls
  `.cancel()` in teardown (after the yield, once the test's loop is already
  closed) raises `RuntimeError: Event loop is closed`. `@loop.before_loop`
  registers a plain coroutine function, not a `Loop` — call it directly
  (`await cog.before_x()`), no `.coro`.
- `cogs/database.py`'s `Session` must be created with `expire_on_commit=False`
  (sessionmaker kwarg) — `_session()` commits and closes before returning, so
  the default would leave every entity `_get_*` returns expired *and*
  detached; the first attribute access anywhere downstream raises
  `sqlalchemy.orm.exc.DetachedInstanceError`. This bit `chess_leaderboard`
  before the fix. To unit-test `database.py` methods without a real MySQL
  server: `_init_engine()` does MySQL-only bootstrap DDL (`SHOW DATABASES
  LIKE`), so monkeypatch `cogs.database.create_engine` to return a fake
  bootstrap engine (whose `.connect()` context manager's `.execute(...)
  .fetchone()` returns a truthy value) for the first call and a real
  `sqlite:///:memory:` engine (with `poolclass=StaticPool,
  connect_args={"check_same_thread": False}` — plain sqlite `:memory:` gives
  each thread its own DB, and `_get_*`/`_save_*` hop threads via
  `asyncio.to_thread`) for the second, then call the real `cog._init_engine()`
  so the exact sessionmaker kwargs under test actually run.
- `DOWNLOAD_DIR` (`cogs/api/youtube.py`) is a single process-wide directory
  shared by every guild's music session. Any cleanup routine that walks that
  directory (`Music.cleanup_files`) must build its "keep" set from *every*
  guild's `GuildMusicState.playlist` (current song + queued songs), not just
  the guild that triggered the cleanup - otherwise a busy guild's active
  download gets deleted by another guild finishing a song. Same class of bug
  as the guild-state-leak this cog was already refactored to fix once; it
  recurred here because `cleanup_files` was missed in that refactor.

  `Btc.sent_message`) is shared across every guild/channel, since discord.py
  cogs are one instance for the whole bot - a second invocation from a
  different channel silently steals the periodic-update target and the
  first channel's embed goes stale forever. Fix: key by channel id in a dict
  (`self.sent_messages[channel.id] = message`), and drop entries whose
  `.edit()` fails (message/channel deleted) instead of leaving them forever.

- Two coroutines can race on a `is_running()`-then-`await
  foo()`-then-`.start()` guard (e.g. `downloader.py::enqueue`) because the
  `await` between the check and the start is a yield point — a second
  invocation can enter the same window before the first calls `.start()`,
  and the second `.start()` raises `RuntimeError`. A same-pattern guard with
  *no* `await` in between (e.g. `state_machine.py::start`, a sync method) is
  not racy — don't flag it as the same bug without checking for an
  intervening await. To reproduce/verify the race in a test: patch the
  awaited method with an `AsyncMock` that blocks on an `asyncio.Event`, fire
  two calls with `asyncio.ensure_future`, `await asyncio.sleep(0)` so both
  reach the block, then release the event and `asyncio.gather` both —
  without a lock this raises `RuntimeError` from the second `.start()`.
